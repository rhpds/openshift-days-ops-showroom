= Secrets Management with HashiCorp Vault

== Module Overview 

**Duration:** 25 minutes +
**Format:** Hands-on lab +
**Audience:** Platform Engineers, Security Engineers, IT Operations

== Learning Objectives

By the end of this module, you will:

* Deploy HashiCorp Vault on OpenShift using the official Helm chart
* Configure Vault policies, Kubernetes authentication, and KV secrets
* Deploy a pod that consumes Vault secrets through the Vault Agent Injector

== Why use Vault on Kubernetes?

HashiCorp Vault stores and brokers access to secrets, certificates, and encryption keys. The official https://github.com/hashicorp/vault-helm[Vault Helm chart] installs Vault on Kubernetes or OpenShift and can optionally deploy the Vault Agent Injector for pod secret injection.

Workshop prerequisites match the upstream chart:

[IMPORTANT]
====
This lab runs Vault in **dev mode** (auto-unsealed, root token `root`). Dev mode is for learning only—data is lost when the Vault pod restarts. Do not use dev mode in production.
====

== Deploy Vault

The workshop install script deploys the HashiCorp Helm chart into the `vault` namespace with the Agent Injector enabled.

.Procedure
. Clone the repo and run the installer from the bastion:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
git clone https://github.com/mfosterrox/openshift-days-ops-showroom.git
cd openshift-days-ops-showroom/setup-scripts
./install-vault.sh
----

The script:

* Adds the HashiCorp Helm repository and installs chart `hashicorp/vault`
* Enables OpenShift integration and **dev mode**
* Creates an OpenShift **Route** for the Vault UI (instead of a cloud LoadBalancer)
* Enables the **Vault Agent Injector** for pod secret injection

Sample output:

[source,bash]
----
[VAULT] Dev mode token (lab only): root
[VAULT] CLI: export VAULT_ADDR="https://vault.apps.<cluster-domain>" VAULT_TOKEN=root
[VAULT] Done.
----

== Verify the deployment

Confirm the Vault server and injector are running:

[source,bash,subs="+macros,+attributes",role=execute]
----
oc get pods,svc,route -n vault
----

You should see the Vault server pod `Running`, the `vault-agent-injector` deployment `Running`, and a Route to the UI.

== Configure Vault

Vault needs a **policy**, **Kubernetes authentication**, and an auth **role** before workloads can read secrets. The upstream guide configures these from inside the Vault pod; this workshop provides a script that performs the same steps from the bastion.

.Procedure
. Set Vault CLI variables on the bastion:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
export VAULT_ADDR="$(oc get route vault -n vault -o jsonpath='https://{.spec.host}')"
export VAULT_TOKEN=root
export VAULT_SKIP_VERIFY=true
cd openshift-days-ops-showroom/setup-scripts
----

[NOTE]
====
`VAULT_SKIP_VERIFY=true` is acceptable for the lab self-signed route certificate. For production, trust the cluster or cert-manager CA instead.
====

. Add the vault variables to the bash session so you can interact with Vault from the bastion:
[source,bash]
----
oc exec -it -n vault vault-0 -- /bin/sh
export VAULT_ADDR=http://127.0.0.1:8200
export VAULT_TOKEN=root

vault policy write read-policy - <<EOF
path "secret*" {
  capabilities = ["read"]
}
EOF

vault auth enable kubernetes
vault write auth/kubernetes/config \
  token_reviewer_jwt="$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)" \
  kubernetes_host="https://kubernetes.default.svc:443" \
  kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

vault write auth/kubernetes/role/vault-role \
  bound_service_account_names=vault-serviceaccount \
  bound_service_account_namespaces=vault \
  policies=read-policy \
  ttl=1h
----

== Create secrets in Vault

The configuration script already created two secrets. You can also create or update them with the CLI or the Vault UI.

. Open the Vault UI in a new browser tab:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
echo "https://$(oc get route vault -n vault -o jsonpath='{.spec.host}')/"
----

. Sign in with method *Token* and token `root`.
. Navigate to *Secrets Engines* → *secret*.
. Use *Create secret* to add paths `login` and `my-first-secret` with the key/value pairs above.

== Access secrets from a Kubernetes pod

With Vault configured and secrets stored, deploy a sample application in the `vault` namespace. The **Vault Agent Injector** mutates the pod to add an init container that authenticates with Kubernetes auth and renders secrets to `/vault/secrets/`.

The deployment uses annotations (from the upstream guide) to inject `secret/login` and `secret/my-first-secret`:

* `vault.hashicorp.com/agent-inject: "true"` — enable injection for this pod
* `vault.hashicorp.com/role: "vault-role"` — Kubernetes auth role to use
* `vault.hashicorp.com/agent-inject-secret-<name>` — Vault path to fetch
* `vault.hashicorp.com/agent-inject-template-<name>` — format written to the volume

.Procedure
. Deploy the service account and test application:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc apply -f demo-vault-app.yaml
oc get pods -n vault -l app=read-vault-secret -w
----

Press `Ctrl+C` when the pod is `2/2` Ready (nginx + Vault Agent init container).

. Read the injected secrets from the pod:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
POD=$(oc get pod -n vault -l app=read-vault-secret -o jsonpath='{.items[0].metadata.name}')
oc exec -n vault "${POD}" -c nginx -- sh -c "cat /vault/secrets/login && echo && cat /vault/secrets/my-first-secret"
----

Expected output:

[source,bash]
----
pattoken=workshop-pattoken-abc123
username=workshop-user
password=workshop-password
----

. Optional: inspect Agent Injector logs if secrets are missing:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc logs -n vault "${POD}" -c vault-agent-init
oc describe pod -n vault "${POD}"
----

. Optional: remove the demo application:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc delete -f demo-vault-app.yaml
----

== What you learned

* **Vault on OpenShift** centralizes secrets outside etcd while the **Agent Injector** delivers them to pods at runtime.
* **Kubernetes auth** ties Vault access to a namespace and service account (`vault-serviceaccount` → role `vault-role`).
* Applications consume files under `/vault/secrets/` without embedding credentials in images or Git.
