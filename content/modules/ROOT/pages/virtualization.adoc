= OpenShift Virtualization

== Module Overview

**Duration:** 30 minutes +
**Audience:** Infrastructure Architects, IT Operations

Your organization has workloads running on VMs that can't be containerized yet — databases, middleware, Windows services. OpenShift Virtualization lets you run those VMs alongside containers on the same platform using KubeVirt — a CNCF project built on RHEL KVM. Same scheduler, same networking, same monitoring, same RBAC, same `oc` commands.

**What this means for your organization:**

* **One platform** — VMs and containers managed with the same tools and skills
* **Migrate at your pace** — run VMs today, containerize when ready. The Migration Toolkit for Virtualization (MTV) handles VM migration with network and storage mapping
* **Production-ready** — live migration, snapshots, SVVP-certified Windows support, bare-metal performance

== Learning Objectives

* Create VMs from YAML and from InstanceTypes
* See how VMs are just pods under the hood
* Expose a VM with Services and Routes — same networking as containers
* Live migrate a running VM between nodes
* Clone a VM for rapid provisioning
* Take snapshots for backup/rollback

== Create a VM

[source,bash,role="execute"]
----
oc new-project vm-demo
----

First, detect the best CPU model that works across all nodes. This ensures live migration works later — nodes may have different CPU generations, and the VM needs a model supported everywhere:

[source,bash,role="execute"]
----
CPU_MODEL=$(oc get nodes -o json | python3 -c "
import json, sys
d = json.load(sys.stdin)
node_models = []
for n in d['items']:
    labels = n['metadata'].get('labels', {})
    models = set(l.replace('cpu-model.node.kubevirt.io/', '') for l in labels if l.startswith('cpu-model.node.kubevirt.io/'))
    node_models.append(models)
common = node_models[0]
for nm in node_models[1:]:
    common = common & nm
epyc = sorted([m for m in common if 'EPYC' in m], reverse=True)
intel = sorted([m for m in common if m.startswith(('Skylake','Broadwell','Haswell','IvyBridge','SandyBridge'))], reverse=True)
print(epyc[0] if epyc else (intel[0] if intel else sorted(common)[-1]))
")
echo "Common CPU model for live migration: $CPU_MODEL"
----

Deploy a RHEL 9 VM from a pre-cached boot image:

[source,bash,role="execute"]
----
cat <<EOF | oc apply -f -
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: rhel9-vm
  namespace: vm-demo
spec:
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: rhel9-vm-disk
    spec:
      sourceRef:
        kind: DataSource
        name: rhel9
        namespace: openshift-virtualization-os-images
      storage:
        resources:
          requests:
            storage: 30Gi
  template:
    metadata:
      labels:
        kubevirt.io/vm: rhel9-vm
    spec:
      domain:
        cpu:
          cores: 1                  # vCPUs allocated to the VM
          model: ${CPU_MODEL}       # common model across all nodes — enables live migration
        devices:
          disks:
          - name: rootdisk
            disk:
              bus: virtio           # virtio = paravirtualized (fast), sata/scsi for legacy OS
          - name: cloudinitdisk
            disk:
              bus: virtio
        resources:
          requests:
            memory: 2Gi             # guaranteed memory — same as pod resource requests
      volumes:
      - name: rootdisk
        dataVolume:
          name: rhel9-vm-disk       # boots from the cloned DataVolume above
      - name: cloudinitdisk
        cloudInitNoCloud:           # injects user/password on first boot
          userData: |
            #cloud-config
            user: cloud-user
            password: redhat
            chpasswd:
              expire: false
EOF
----

Wait for the VM to start (1-2 minutes):

[source,bash,role="execute"]
----
oc get vms -w
----

Press `Ctrl+C` when STATUS shows `Running`.

=== InstanceTypes: Cloud-Style Provisioning

The YAML above gives you full control, but in practice most teams don't want to write VM specs from scratch. OpenShift Virtualization provides **InstanceTypes** — predefined compute profiles similar to cloud instance families:

[cols="1,2"]
|===
|Series |Use case

|**U** (Universal) |General-purpose workloads
|**O** (Overcommitted) |Dev/test with CPU overcommit
|**CX** (Compute) |CPU-intensive workloads
|**M** (Memory) |Memory-intensive (databases, caching)
|**N** (Network) |Network-intensive workloads
|===

View available InstanceTypes:

[source,bash,role="execute"]
----
oc get virtualmachineclusterinstancetype | head -10
----

This is the same experience your teams get in AWS/Azure — pick a size, pick a boot image, deploy. No YAML needed. In the OpenShift console, you can create VMs by selecting an InstanceType and a boot source from `openshift-virtualization-os-images` with a few clicks.

=== Serial Console (CLI)

First, install the `virtctl` CLI tool (downloaded directly from the cluster):

[source,bash,role="execute"]
----
VIRTCTL_URL=$(oc get consoleclidownload virtctl-clidownloads-kubevirt-hyperconverged -o jsonpath='{.spec.links[?(@.text=="Download virtctl for Linux for x86_64")].href}')
curl -sL "$VIRTCTL_URL" | tar xz -C /tmp && sudo mv /tmp/virtctl /usr/local/bin/
virtctl version --client
----

Now connect to the VM's serial console:

[source,bash,role="execute"]
----
virtctl console rhel9-vm
----

Login with `cloud-user` / `redhat`. Type `exit` to disconnect. After exiting, click the **reload button** (circular arrow) at the top next to the terminal tab to reconnect as the admin user.

== It's Just a Pod

This is the key insight for operations teams. Your VM runs inside a `virt-launcher` pod — which means everything you already know about managing pods applies to VMs. Same scheduler, same RBAC, same resource quotas, same `oc` commands. You don't need separate tools, separate teams, or separate runbooks for VMs vs containers:

[source,bash,role="execute"]
----
oc get pods -n vm-demo
----

You'll see:

----
NAME                          READY   STATUS    RESTARTS   AGE
virt-launcher-rhel9-vm-xxxxx   2/2     Running   0          3m
----

That's your VM. It's a pod. The same `oc` commands work:

[source,bash,role="execute"]
----
oc describe pod -l kubevirt.io/vm=rhel9-vm -n vm-demo | tail -10
----

== Expose the VM (Same Networking as Containers)

VMs get pod IPs on the same SDN as containers — same Services, same Routes, same NetworkPolicies. One networking model for VMs and containers:

[source,bash,role="execute"]
----
oc get vmi rhel9-vm -o jsonpath='{.status.interfaces[0].ipAddress}' && echo ""
----

Create a Service pointing to the VM:

[source,bash,role="execute"]
----
oc apply -f - <<EOF
apiVersion: v1
kind: Service
metadata:
  name: rhel9-vm-ssh
  namespace: vm-demo
spec:
  selector:
    kubevirt.io/vm: rhel9-vm
  ports:
  - port: 22
    targetPort: 22
EOF
----

[source,bash,role="execute"]
----
oc get svc rhel9-vm-ssh -n vm-demo
----

The VM is now reachable via the service — the same way you'd expose a container. Same label selectors, same service types, same networking model.

You can also create a Route to expose VM-hosted applications externally — exactly like you would for a container:

[source,bash,role="execute"]
----
oc expose svc rhel9-vm-ssh --port=22 --name=rhel9-vm-route -n vm-demo 2>/dev/null || echo "Route creation skipped (SSH is not HTTP-routable, but the pattern is the same for HTTP services)"
----

For VMs running web applications (Apache, IIS, Tomcat), you'd create a Service on port 80/443 and a Route with TLS edge termination — the same workflow you use for containers. When migrating multi-tier applications, move the VMs with MTV, wire them up with Services and Routes, and they're live on OpenShift without changing the application.

== Live Migration

This is the feature that makes OpenShift Virtualization production-ready. When you need to patch a node, upgrade the OS, or replace hardware, you drain the node. Containers reschedule automatically, and VMs live migrate automatically too. `oc adm drain` handles both. Zero downtime during maintenance windows.

Check which node the VM is currently on:

[source,bash,role="execute"]
----
oc get vmi rhel9-vm -o jsonpath='{.status.nodeName}' && echo ""
----

Trigger a live migration:

[source,bash,role="execute"]
----
virtctl migrate rhel9-vm
----

In the **Fleet Virtualization** console, click **rhel9-vm** and watch the **Migrations** tab — it shows migration progress with phases (Scheduling → TargetReady → Running → Succeeded). The **Overview** tab updates the node name when the migration completes.

You can also watch via CLI:

[source,bash,role="execute"]
----
oc get vmim -w
----

Press `Ctrl+C` when the migration completes. Verify the VM moved to a different node:

[source,bash,role="execute"]
----
oc get vmi rhel9-vm -o jsonpath='{.status.nodeName}' && echo ""
----

The VM moved to a different node while still running. No downtime, no reconnection needed.

== Clone a VM

Need another copy of a VM — for testing, dev environments, or scaling out? Clone it. This creates a full copy of the VM's disk and configuration:

[source,bash,role="execute"]
----
cat <<EOF | oc apply -f -
apiVersion: clone.kubevirt.io/v1alpha1
kind: VirtualMachineClone
metadata:
  name: rhel9-vm-clone
  namespace: vm-demo
spec:
  source:
    apiGroup: kubevirt.io
    kind: VirtualMachine
    name: rhel9-vm
  target:
    apiGroup: kubevirt.io
    kind: VirtualMachine
    name: rhel9-vm-clone
EOF
----

Wait for the clone to complete:

[source,bash,role="execute"]
----
oc get vmclone rhel9-vm-clone -n vm-demo -w
----

Press `Ctrl+C` when the PHASE shows `Succeeded`, then check your VMs:

[source,bash,role="execute"]
----
oc get vms -n vm-demo
----

You'll see both `rhel9-vm` and `rhel9-vm-clone`. The clone has its own independent disk — changes to one don't affect the other. This is how you'd create golden images: configure a VM exactly how you want it, then clone it for each environment.

== Snapshots

Every operations team has been burned by a patch that broke something. Snapshots give you a safety net — capture the VM's disk and configuration before a change, and roll back in seconds if it goes wrong. These are Kubernetes-native resources that you can manage with `oc`, automate with pipelines, and back up with OADP.

Create a snapshot of your running VM:

[source,bash,role="execute"]
----
cat <<EOF | oc apply -f -
apiVersion: snapshot.kubevirt.io/v1beta1
kind: VirtualMachineSnapshot
metadata:
  name: rhel9-vm-snap-1
  namespace: vm-demo
spec:
  source:
    apiGroup: kubevirt.io
    kind: VirtualMachine
    name: rhel9-vm
EOF
----

Check snapshot status:

[source,bash,role="execute"]
----
oc get vmsnapshot
----

Wait for `readyToUse: true` — this means the snapshot has finished and is available for restore. You can also see snapshots in the **Fleet Virtualization** console: click **rhel9-vm** -> **Snapshots** tab:

image::vm-snapshot-ready.png[VM Snapshots tab showing rhel9-vm-snap-1 with Ready status]

=== Restore from Snapshot

**The scenario:** Imagine you've just applied an OS patch or a configuration change to this VM and something broke — the application won't start, the network config is wrong, or a package update caused a regression. Instead of spending hours troubleshooting or rebuilding from scratch, you restore to the snapshot you took before the change. The VM comes back exactly as it was — disk, configuration, everything.

This is the same workflow you'd use in production: snapshot before maintenance, verify the change works, delete the snapshot if it's good, or restore if it's not.

Stop the VM first, then apply the restore:

[source,bash,role="execute"]
----
virtctl stop rhel9-vm
oc wait --for=jsonpath='{.status.printableStatus}'=Stopped vm/rhel9-vm --timeout=60s
----

[source,bash,role="execute"]
----
cat <<EOF | oc apply -f -
apiVersion: snapshot.kubevirt.io/v1beta1
kind: VirtualMachineRestore
metadata:
  name: rhel9-vm-restore
  namespace: vm-demo
spec:
  target:
    apiGroup: kubevirt.io
    kind: VirtualMachine
    name: rhel9-vm
  virtualMachineSnapshotName: rhel9-vm-snap-1
EOF
----

[source,bash,role="execute"]
----
oc get vmrestore
----

Start the VM again:

[source,bash,role="execute"]
----
virtctl start rhel9-vm
oc get vms -w
----

Press `Ctrl+C` when `Running`. The VM's disk and configuration are now back to the exact state they were in when you took the snapshot. This is how you'd roll back a failed OS patch or a broken config change — without rebuilding the VM from scratch.

== Cleanup

[source,bash,role="execute"]
----
# Stop VMs, then delete the namespace (removes all VMs, snapshots, clones, services)
virtctl stop rhel9-vm -n vm-demo 2>/dev/null || true
virtctl stop rhel9-vm-clone -n vm-demo 2>/dev/null || true
oc delete namespace vm-demo --ignore-not-found

echo "Cleanup complete"
----
