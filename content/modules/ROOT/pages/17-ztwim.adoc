= Zero-Trust Database Access with SPIFFE mTLS

== Module Overview

**Duration:** 45 minutes +
**Format:** Hands-on lab +
**Audience:** Security Engineers, Platform Engineers, IT Operations

== Learning Objectives

By the end of this module, you will:

* Configure ZTWIM for the lab using the workshop setup scripts
* Deploy PostgreSQL with SPIFFE-issued certificates and a separate client workload identity
* Establish mutual TLS (mTLS) database connections using short-lived SPIFFE certificates instead of passwords
* Inspect certificate properties, identity binding, and automatic certificate rotation
* Confirm that connections without valid certificates are rejected

== Why SPIFFE mTLS for workloads?

Password-based database authentication creates persistent credential theft targets. Compromised passwords enable lateral movement, and stolen credentials remain valid until manually rotated. Network segmentation alone cannot prevent authenticated attackers from accessing databases.

Zero-trust principles require cryptographic proof of workload identity—not just network proximity or static secrets. This lab demonstrates how ZTWIM issues short-lived X.509 certificates bound to Kubernetes workload identity (namespace + service account), enabling PostgreSQL to authenticate clients via mutual TLS with no passwords stored in Secrets or ConfigMaps.

[cols="1,2,2",options="header"]
|===
|Area | Risk If Ignored | Analogy
|Static Database Passwords | Credential theft enables persistence | Master key copies floating everywhere
|Password-Based Auth | Brute force / spray attacks | Guessable combination lock
|Network-Only Security | Lateral movement after breach | Locked front door, open internal doors
|Manual Cert Rotation | Stale credentials remain valid | Expired building access badges still work
|No Mutual Authentication | Server cannot verify client | One-way ID check at secure facility
|===

== Inspect server certificate properties

SPIFFE certificates are automatically generated and mounted at `/opt/postgresql-certs/` by the `spiffe-helper` sidecar.

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe -c postgresql-spiffe deployment/postgresql-spiffe \
  cat /opt/postgresql-certs/svid.pem | openssl x509 -noout -text
----

=== Key certificate fields

*Subject (Certificate Identity)*::
The Common Name identifies the server hostname:
+
[source,text]
----
Subject: C=US, O=SPIRE, CN=postgresql-spiffe.postgresql-spiffe.svc
----

*Validity Period (Short-Lived)*::
SPIFFE certificates have intentionally short lifespans (default: 1 hour):
+
[source,text]
----
Validity:
    Not Before: Apr 22 12:32:55 2026 GMT
    Not After : Apr 22 13:32:55 2026 GMT
----

*Subject Alternative Name (SAN)*::
Contains the SPIFFE ID URI that uniquely identifies this workload:
+
[source,text]
----
X509v3 Subject Alternative Name:
    DNS:postgresql-spiffe.postgresql-spiffe.svc
    URI:spiffe://<trust-domain>/ns/postgresql-spiffe/sa/postgresql-spiffe
----

The SPIFFE ID format: `spiffe://<trust-domain>/ns/<namespace>/sa/<serviceaccount>`

=== View certificate files

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe -c postgresql-spiffe deployment/postgresql-spiffe \
  ls -la /opt/postgresql-certs/
----

Expected output:

[source,text]
----
-rw-------. 1 postgres postgres  241 svid.key         ← Private key (600 permissions)
-rw-r--r--. 1 postgres postgres 1298 svid.pem         ← Public certificate
-rw-r--r--. 1 postgres postgres 1403 svid_bundle.pem  ← CA bundle for verification
----

== Inspect client certificate properties

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  cat /opt/postgresql-certs/svid.pem | openssl x509 -noout -text
----

The client certificate CN must match a valid PostgreSQL database user:

[source,text]
----
Subject: C=US, O=SPIRE, CN=postgresql_spiffe
----

Each workload gets a unique SPIFFE ID based on its Kubernetes identity:

* *Server*: `spiffe://.../ns/postgresql-spiffe/sa/postgresql-spiffe`
* *Client*: `spiffe://.../ns/postgresql-spiffe-client/sa/postgresql-spiffe-client`

Check certificate expiration:

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  sh -c 'echo "Current time: $(date)"; openssl x509 -in /opt/postgresql-certs/svid.pem -noout -dates'
----

The `spiffe-helper` sidecar automatically fetches a new certificate approximately 30 minutes before expiration.

== Establish mTLS connection from client

Connect to PostgreSQL using mutual TLS authentication with SPIFFE certificates—no password required.

=== Connect via one-line command

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  psql "host=postgresql-spiffe.postgresql-spiffe.svc port=5432 user=postgresql_spiffe dbname=testdb sslmode=verify-full sslcert=/opt/postgresql-certs/svid.pem sslkey=/opt/postgresql-certs/svid.key sslrootcert=/opt/postgresql-certs/svid_bundle.pem" \
  -c "SELECT current_user, current_database();"
----

Expected output:

[source,text]
----
   current_user    | current_database
-------------------+------------------
 postgresql_spiffe | testdb
(1 row)
----

Success—the connection authenticated using certificates with no password.

=== Connection string parameters

* `host=postgresql-spiffe.postgresql-spiffe.svc` — Kubernetes service DNS name
* `port=5432` — Standard PostgreSQL port
* `user=postgresql_spiffe` — Must match the certificate CN
* `dbname=testdb` — Database created by the initialization script
* `sslmode=verify-full` — Strictest SSL mode: server certificate verified, hostname matched, client certificate required
* `sslcert`, `sslkey`, `sslrootcert` — Paths to SPIFFE-issued client certificate, private key, and CA bundle

== Test security controls: connection without certificate

To prove certificate-based authentication is enforced, attempt a connection without client certificates:

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  psql "host=postgresql-spiffe.postgresql-spiffe.svc port=5432 user=postgresql_spiffe dbname=testdb sslmode=require" \
  2>&1 | head -5
----

Expected error:

[source,text]
----
psql: error: connection to server at "postgresql-spiffe.postgresql-spiffe.svc" failed:
FATAL: connection requires a valid client certificate
----

This demonstrates:

* Network connectivity alone is insufficient—the workload must present a valid SPIFFE certificate
* No password bypass is possible because no passwords exist for the `postgresql_spiffe` user
* PostgreSQL `pg_hba.conf` requires SSL *and* valid client certificates (`hostssl ... cert`)

=== Test from unauthorized pod (advanced)

[source,bash,subs="+macros,+attributes",role=execute]
----
oc run unauthorized-client --image=registry.redhat.io/rhel9/postgresql-16:latest \
  -n postgresql-spiffe-client --command -- sleep infinity
oc wait --for=condition=Ready pod/unauthorized-client -n postgresql-spiffe-client --timeout=60s
oc exec -n postgresql-spiffe-client unauthorized-client -- \
  psql "host=postgresql-spiffe.postgresql-spiffe.svc port=5432 user=postgresql_spiffe dbname=testdb" \
  2>&1 | head -3
oc delete pod unauthorized-client -n postgresql-spiffe-client
----

Even pods within the same namespace cannot connect without valid SPIFFE certificates.

== Observe automatic certificate rotation (optional)

SPIFFE certificates rotate before expiration with zero downtime.

. Capture the current certificate serial number:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  sh -c 'openssl x509 -in /opt/postgresql-certs/svid.pem -noout -serial -enddate'

+
. Monitor `spiffe-helper` logs for rotation events:

[source,bash,subs="+macros,+attributes",role=execute]

oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \

  -c "SELECT 'Connection successful after rotation' AS status;"
----

== What you learned

* *ZTWIM* on this cluster already provides SPIRE server, agents, CSI driver, and OIDC discovery for workload identity.
* *SPIFFE workload identity* binds X.509 certificates to Kubernetes namespace and service account, replacing static database passwords.
* *Mutual TLS* enforces bidirectional certificate verification—both client and server prove cryptographic identity.
* *Short-lived certificates* (approximately 1 hour) limit exposure windows and rotate automatically via `spiffe-helper` sidecars.
* *Zero-trust enforcement*: connections without valid SPIFFE certificates are rejected regardless of network access.

[cols="1,2,2,2",options="header"]
|===
|Aspect | Password Auth | Certificate Auth (SPIFFE) | Mutual TLS (This Lab)
|Credential Type | Static password | Client certificate only | Client + server certificates
|Rotation Burden | Manual | Automatic | Automatic (both sides)
|Theft Impact | Persistent access | Time-limited exposure | Time-limited + verified server
|Identity Binding | None (password reuse) | Certificate CN | Certificate CN + SPIFFE ID
|===
