= Secrets Management with HashiCorp Vault

== Module Overview 

**Duration:** 25 minutes +
**Format:** Hands-on lab +
**Audience:** Platform Engineers, Security Engineers, IT Operations

== Learning Objectives

By the end of this module, you will:

* Deploy HashiCorp Vault on OpenShift using the official Helm chart
* Configure Vault policies, Kubernetes authentication, and KV secrets
* Deploy a pod that consumes Vault secrets through the Vault Agent Injector
* Deploy the External Secrets Operator to fetch secrets from Vault and inject them into Kubernetes Secrets

== Why use Vault on Kubernetes?

HashiCorp Vault stores and brokers access to secrets, certificates, and encryption keys. Instead of embedding credentials in container images, Git repositories, or long-lived Kubernetes `Secret` objects, Vault keeps sensitive data in a dedicated secrets store and issues short-lived access on demand. The official https://github.com/hashicorp/vault-helm[Vault Helm chart] installs Vault on Kubernetes or OpenShift and can optionally deploy the Vault Agent Injector for pod secret injection. The injector mutates pod specifications at admission time so an init container can authenticate to Vault and render secrets into a volume before your application starts. 

In this module you will also use External Secrets Operator as a complementary pattern that syncs Vault (or your preferred secrets storage solution) into native Kubernetes `Secret` resources for workloads that expect standard secret references.

[NOTE]
====
This lab runs Vault in **dev mode** (auto-unsealed, root token `root`). Dev mode is for learning only—data is lost when the Vault pod restarts. Do not use dev mode in production.
====

== Configure Vault

In this step, we will configure Vault policies and authentication methods to securely manage and access secrets within the Kubernetes cluster. This configuration ensures that only authorized applications can retrieve sensitive data from Vault.

You will start by using the CLI to configure Vault policies and authentication methods and take a look at the UI to verify the configuration.

.Procedure
. Connect to the Vault pod:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc exec -it -n vault vault-0 -- /bin/sh
export VAULT_ADDR=http://127.0.0.1:8200
export VAULT_TOKEN=root
----

. Create a policy file that allows a pod to read secrets:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
cat <<EOF > /home/vault/read-policy.hcl
path "secret*" {
  capabilities = ["read"]
}
EOF
----

. Apply the policy:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
vault policy write read-policy /home/vault/read-policy.hcl
----
+
. Enable Kubernetes authentication:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
vault auth enable kubernetes
----
+
. Configure Vault to communicate with the Kubernetes API server:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
vault write auth/kubernetes/config \
  token_reviewer_jwt="$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)" \
  kubernetes_host="https://kubernetes.default.svc:443" \
  kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt
----
+
. Enable the KV v2 secrets engine at `secret/` (skip if already enabled):
+
[source,bash,subs="+macros,+attributes",role=execute]
----
vault secrets enable -path=secret kv-v2 2>/dev/null || true
----
+
. Create role `vault-role` that binds the policy to service account `vault-serviceaccount` in namespace `vault`:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
AUD=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token | cut -d. -f2 | base64 -d 2>/dev/null | sed -n 's/.*"aud":\["\([^"]*\)".*/\1/p')
vault write auth/kubernetes/role/vault-role \
  bound_service_account_names=vault-serviceaccount \
  bound_service_account_namespaces=vault \
  policies=read-policy \
  ttl=1h \
  audience="${AUD}"
----
+
Vault 1.21 validates the service account token `aud` claim against the role `audience`. The command above reads the cluster audience from the Vault pod token (typically `https://kubernetes.default.svc` on OpenShift). Do **not** use `audience=vault`—that value does not match pod tokens and causes `403 invalid audience` errors in the Agent Injector init container.
+
If you previously created `vault-role` with the wrong audience, delete it before recreating:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
vault delete auth/kubernetes/role/vault-role
----
+
Then run the `vault write auth/kubernetes/role/vault-role` command again.
+
If you omit `audience` entirely, Vault 1.20 and later may print a warning when the role is created:
+
[source,bash]
----
WARNING! The following warnings were returned from Vault:

  * Role vault-role does not have an audience configured. While audiences are
  not required, consider specifying one if your use case would benefit from
  additional JWT claim verification.
----
+
[NOTE]
====
Audience warnings on role creation are **not errors**—the role was created successfully. Setting `audience` to the cluster API audience (as in the command above) satisfies Vault 1.21 validation and avoids both warnings and login failures.
====
+
. Exit the Vault pod shell when configuration is complete:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
exit
----

== Access secrets from a Kubernetes pod

With Vault configured and secrets stored, you will deploy a sample application in the `vault` namespace. The **Vault Agent Injector** mutates the pod to add an init container that authenticates with Kubernetes auth and renders secrets to `/vault/secrets/`.

The deployment uses annotations to inject `secret/login` and `secret/my-first-secret`:

* `vault.hashicorp.com/agent-inject: "true"` — enable injection for this pod
* `vault.hashicorp.com/role: "vault-role"` — Kubernetes auth role to use
* `vault.hashicorp.com/agent-inject-secret-<name>` — Vault path to fetch
* `vault.hashicorp.com/agent-inject-template-<name>` — format written to the volume

.Procedure
. Deploy the service account and test application:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
cat <<'EOF' | oc apply -f -
---
apiVersion: v1
kind: ServiceAccount
metadata:
  name: vault-serviceaccount
  namespace: vault
  labels:
    app: read-vault-secret
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vault-test
  namespace: vault
  labels:
    app: read-vault-secret
spec:
  replicas: 1
  selector:
    matchLabels:
      app: read-vault-secret
  template:
    metadata:
      labels:
        app: read-vault-secret
      annotations:
        vault.hashicorp.com/agent-inject: "true"
        vault.hashicorp.com/agent-inject-status: "update"
        vault.hashicorp.com/role: "vault-role"
        vault.hashicorp.com/agent-inject-address: "http://vault.vault.svc:8200"
        vault.hashicorp.com/agent-inject-secret-login: "secret/login"
        vault.hashicorp.com/agent-inject-template-login: |
          {{- with secret "secret/login" -}}
          rhos-token={{ index .Data.data "rhos-token" }}
          {{- end }}
        vault.hashicorp.com/agent-inject-secret-my-first-secret: "secret/my-first-secret"
        vault.hashicorp.com/agent-inject-template-my-first-secret: |
          {{- with secret "secret/my-first-secret" -}}
          username={{ .Data.data.username }}
          password={{ .Data.data.password }}
          {{- end }}
    spec:
      serviceAccountName: vault-serviceaccount
      containers:
        - name: nginx
          image: registry.access.redhat.com/ubi9/nginx-124:latest
          ports:
            - containerPort: 8080
EOF
----
+
. Wait for the pod to be ready:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc get pods -n vault -l app=read-vault-secret -w
----
+
Press `Ctrl+C` when the pod is `2/2` Ready (nginx + Vault Agent init container).

. Read the injected secrets from the pod:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
POD=$(oc get pod -n vault -l app=read-vault-secret -o jsonpath='{.items[0].metadata.name}')
oc exec -n vault "${POD}" -c nginx -- sh -c "cat /vault/secrets/login && echo && cat /vault/secrets/my-first-secret"
----
+
Expected output:
+
[source,bash]
----
rhos-token=workshop-token-abc123
username=workshop-user
password=workshop-password
----
+
. Optional: inspect Agent Injector logs if secrets are missing:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc logs -n vault "${POD}" -c vault-agent-init
oc describe pod -n vault "${POD}"
----
+
. Optional: remove the demo application:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc delete deployment vault-test serviceaccount vault-serviceaccount -n vault
----

== Access secrets with External Secrets Operator

The first deployment used the **Vault Agent Injector** to render secrets as files under `/vault/secrets/`. The **External Secrets Operator (ESO)** offers a different pattern: it reads secrets from Vault and materializes them as native Kubernetes `Secret` objects. Workloads then consume those secrets through `env`, `envFrom`, or volume mounts—the same way they would use any other Kubernetes secret.

=== Install External Secrets Operator

.Procedure
. Install ESO from the official Helm chart into the `external-secrets` namespace:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
helm repo add external-secrets https://charts.external-secrets.io
helm repo update external-secrets
helm upgrade --install external-secrets external-secrets/external-secrets \
  -n external-secrets \
  --create-namespace \
  --set installCRDs=true \
  --wait --timeout 10m
----

. Confirm the operator is running:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc get pods -n external-secrets
----

You should see the `external-secrets` deployment pod in `Running` state.

=== Store the second secret in Vault

ESO syncs the Vault path `secret/my-second-secret` into a Kubernetes `Secret` named `my-second-secret-sync`. You should have created `my-second-secret` in the Vault UI with both `username` and `password` keys.

If you created the path with only `username`, or skipped the UI step, add the missing key from the bastion:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc exec -n vault vault-0 -- env VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=root \
  vault kv put secret/my-second-secret username=workshop-user-2 password=workshop-password-2
----
+
Verify both keys exist before continuing:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc exec -n vault vault-0 -- env VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=root \
  vault kv get secret/my-second-secret
----

In dev mode, ESO authenticates to Vault with the root token stored in a Kubernetes `Secret`. A `SecretStore` defines how to reach Vault; an `ExternalSecret` defines which Vault paths to copy and the name of the target Kubernetes `Secret`.

.Procedure
. Apply the token, `SecretStore`, and `ExternalSecret` in the `vault` namespace:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
cat <<'EOF' | oc apply -f -
---
apiVersion: v1
kind: Secret
metadata:
  name: vault-token
  namespace: vault
type: Opaque
stringData:
  token: root
---
apiVersion: external-secrets.io/v1
kind: SecretStore
metadata:
  name: vault-backend
  namespace: vault
spec:
  provider:
    vault:
      server: "http://vault.vault.svc:8200"
      path: "secret"
      version: "v2"
      auth:
        tokenSecretRef:
          name: vault-token
          key: token
---
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: sync-my-second-secret
  namespace: vault
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: SecretStore
  target:
    name: my-second-secret-sync
    creationPolicy: Owner
  data:
    - secretKey: username
      remoteRef:
        key: my-second-secret
        property: username
    - secretKey: password
      remoteRef:
        key: my-second-secret
        property: password
EOF
----

. Verify ESO created the Kubernetes `Secret`:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc get externalsecret,secretstore -n vault
oc get externalsecret sync-my-second-secret -n vault -o jsonpath='{.status.conditions[?(@.type=="Ready")].status}{"\n"}'
oc get secret my-second-secret-sync -n vault -o jsonpath='{.data.username}{"\n"}' | base64 -d && echo
----

Expected: ExternalSecret `Ready` status is `True`, and the decoded username is `workshop-user-2`.

If status is `SecretSyncedError` and `my-second-secret-sync` was not created, check that Vault path `my-second-secret` contains **both** `username` and `password` keys (the UI step previously only added `username`). Inspect the error, fix the Vault secret, and ESO will retry:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc describe externalsecret sync-my-second-secret -n vault | tail -20
oc exec -n vault vault-0 -- env VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=root \
  vault kv put secret/my-second-secret username=workshop-user-2 password=workshop-password-2
oc get externalsecret sync-my-second-secret -n vault -w
----
+
Press `Ctrl+C` when `READY` is `True`, then re-run the verification commands above.

=== Deploy a second application using the synced Secret

Deploy a second workload that reads credentials from the Kubernetes `Secret` ESO created—not from Vault Agent files.

.Procedure
. Deploy the application:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
cat <<'EOF' | oc apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: eso-vault-test
  namespace: vault
  labels:
    app: eso-read-vault-secret
spec:
  replicas: 1
  selector:
    matchLabels:
      app: eso-read-vault-secret
  template:
    metadata:
      labels:
        app: eso-read-vault-secret
    spec:
      containers:
        - name: nginx
          image: registry.access.redhat.com/ubi9/nginx-124:latest
          ports:
            - containerPort: 8080
          env:
            - name: USERNAME
              valueFrom:
                secretKeyRef:
                  name: my-second-secret-sync
                  key: username
            - name: PASSWORD
              valueFrom:
                secretKeyRef:
                  name: my-second-secret-sync
                  key: password
EOF
----

[source,bash,subs="+macros,+attributes",role=execute]
----
oc get pods -n vault -l app=eso-read-vault-secret -w
----

Press `Ctrl+C` when the pod is `1/1` Ready.

. Confirm the pod received the synced credentials:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
ESO_POD=$(oc get pod -n vault -l app=eso-read-vault-secret -o jsonpath='{.items[0].metadata.name}')
oc exec -n vault "${ESO_POD}" -c nginx -- sh -c 'echo "username=${USERNAME}" && echo "password=${PASSWORD}"'
----

Expected output:

[source,bash]
----
username=workshop-user-2
password=workshop-password-2
----

. Optional: remove the ESO demo resources:
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc delete deployment eso-vault-test -n vault
oc delete externalsecret/sync-my-second-secret secretstore/vault-backend -n vault
oc delete secret vault-token my-second-secret-sync -n vault
----

== What you learned

* **Vault on OpenShift** centralizes secrets outside etcd while the **Agent Injector** delivers them to pods at runtime.
* **Kubernetes auth** ties Vault access to a namespace and service account (`vault-serviceaccount` → role `vault-role`).
* The **Vault Agent Injector** writes secrets to files under `/vault/secrets/` inside the pod.
* **External Secrets Operator** syncs Vault data into native Kubernetes `Secret` objects for use with `env`, `envFrom`, or volume mounts.
* You deployed two consumption patterns: Agent Injector (`vault-test`) and ESO-backed Kubernetes Secrets (`eso-vault-test`).
