== Learning Objectives

By the end of this module, you will:

* Install the Red Hat Zero Trust Workload Identity Manager (ZTWIM) operator and SPIRE components
* Deploy PostgreSQL with SPIFFE-issued certificates for mutual TLS (mTLS)
* Connect a client workload using certificate-based authentication (no passwords)
* Verify that connections without valid SPIFFE certificates are rejected

== Why cryptographic workload identity?

Password-based database authentication creates persistent credential theft targets. Compromised passwords enable lateral movement, and stolen credentials remain valid until manually rotated. Zero-trust principles require cryptographic proof of workload identity, not just network proximity or static secrets.

The Zero Trust Workload Identity Manager is an OpenShift Day-2 operator (provided by Red Hat) that automates SPIFFE/SPIRE components: `spire-server`, `spire-agent`, `spiffe-csi-driver`, and `oidc-discovery-provider`. It issues short-lived, verifiable workload identities instead of long-lived credentials.

Project reference: link:https://github.com/openshift/zero-trust-workload-identity-manager[zero-trust-workload-identity-manager on GitHub^]

This lab follows the link:https://github.com/SeanRickerd/Roadshow-ZTWIM[Roadshow-ZTWIM PostgreSQL SPIFFE mTLS exercise^].

== Inspect SPIFFE certificates

SPIFFE certificates are mounted at `/opt/postgresql-certs/` by the spiffe-helper sidecar.

=== Server certificate

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe -c postgresql-spiffe deployment/postgresql-spiffe \
  cat /opt/postgresql-certs/svid.pem | openssl x509 -noout -text
----

Key fields to examine:

* *Subject CN*: `postgresql-spiffe.postgresql-spiffe.svc` (server hostname)
* *Issuer*: SPIRE CA (`CN=redhat.com`)
* *Validity*: short-lived (approximately 1 hour)
* *SAN URI*: `spiffe://<trust-domain>/ns/postgresql-spiffe/sa/postgresql-spiffe`

=== Client certificate

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  cat /opt/postgresql-certs/svid.pem | openssl x509 -noout -text
----

The client certificate CN is `postgresql_spiffe`, which matches the PostgreSQL database user created by the initialization script. PostgreSQL maps the certificate CN directly to the database username during certificate authentication.

== Establish an mTLS connection

Connect to PostgreSQL using mutual TLS with SPIFFE certificates and no password.

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  psql "host=postgresql-spiffe.postgresql-spiffe.svc port=5432 user=postgresql_spiffe dbname=testdb sslmode=verify-full sslcert=/opt/postgresql-certs/svid.pem sslkey=/opt/postgresql-certs/svid.key sslrootcert=/opt/postgresql-certs/svid_bundle.pem" \
  -c "SELECT current_user, current_database();"
----

Expected output:

[source,text]
----
   current_user    | current_database
-------------------+------------------
 postgresql_spiffe | testdb
(1 row)
----

The connection authenticated using certificates only. No password was required.

=== Connection string parameters

* `sslmode=verify-full` — strictest mode; verifies server certificate, hostname, and requires a client certificate
* `sslcert` / `sslkey` — client SPIFFE certificate and private key
* `sslrootcert` — CA bundle for verifying the server certificate
* `user=postgresql_spiffe` — must match the client certificate CN

== Verify certificate-based authentication

=== Insert and query test data

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  psql "host=postgresql-spiffe.postgresql-spiffe.svc port=5432 user=postgresql_spiffe dbname=testdb sslmode=verify-full sslcert=/opt/postgresql-certs/svid.pem sslkey=/opt/postgresql-certs/svid.key sslrootcert=/opt/postgresql-certs/svid_bundle.pem" <<'SQL'
INSERT INTO test_table (name, text)
VALUES ('spiffe-test', 'Authenticated with certificates!');
SELECT * FROM test_table;
SQL
----

=== Connection without a certificate (should fail)

Attempting to connect without a client certificate proves that certificate auth is enforced:

[source,bash,subs="+macros,+attributes",role=execute]
----
oc rsh -n postgresql-spiffe-client -c postgresql-spiffe-client deployment/postgresql-spiffe-client \
  psql "host=postgresql-spiffe.postgresql-spiffe.svc port=5432 user=postgresql_spiffe dbname=testdb sslmode=require" \
  2>&1 | head -5
----

The connection should fail with an error indicating a valid client certificate is required. PostgreSQL `pg_hba.conf` is configured for `cert` authentication with no password fallback.

== What you learned

* **ZTWIM** automates SPIFFE/SPIRE on OpenShift, issuing short-lived workload identities bound to Kubernetes namespace and service account.
* **ClusterSPIFFEID** resources define which certificates each workload receives.
* **mTLS with SPIFFE** replaces static database passwords with cryptographic proof of identity on both client and server.
* **Automatic rotation** via spiffe-helper sidecars keeps certificates fresh without manual intervention.
