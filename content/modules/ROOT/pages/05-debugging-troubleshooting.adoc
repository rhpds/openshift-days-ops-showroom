= Debugging & Troubleshooting
:page-toclevels: 1

== Getting Started

=== Module Overview

**Duration:** 10-15 minutes +
**Format:** Hands-on Lab - Real Problem Solving +
**Audience:** IT Operations, SRE, Platform Engineers

Your cluster is running a weather application made up of several microservices. Some of them have just broken. Your job is to diagnose and fix each one - these are the same kinds of issues that generate real support tickets.

=== Learning Objectives

By the end of this module, you will be able to:

* Read pod events and logs to identify root causes
* Fix ImagePullBackOff, CrashLoopBackOff, ConfigMap errors, resource constraint violations, service routing issues, and SCC permission failures
* Use the Observe → Diagnose → Fix → Verify workflow

=== Your Diagnostic Toolkit

These are the commands you'll use constantly:

[source,bash,role="copypaste"]
----
# What's broken?
oc get pods -n <namespace>

# Why is it broken?
oc describe pod <pod-name> -n <namespace>

# What did the app say before it died?
oc logs <pod-name> -n <namespace>
oc logs <pod-name> -n <namespace> --previous

# Check deployment events (for pods that never got created)
oc describe deployment <deployment-name> -n <namespace>
----

== Fix the Problems

=== Problem 1: ImagePullBackOff (weather-frontend)

**Diagnose**

In the **OCP Console**, click on the `weather-frontend` pod. The **Events** tab at the bottom shows the error with colour-coded severity - you'll see the image pull failure immediately.

image::debugging-frontend-events.png[weather-frontend pod Events tab showing ImagePullBackOff error]

You can also check via CLI:

[source,bash,role="execute"]
----
oc describe pod -l app=weather-frontend -n debug-lab | tail -10
----

Look at the **Events** section. You should see errors like:

----
Warning  Failed   Failed to pull image "registry.access.redhat.com/ubi9/httpd-24:9.9": ... manifest unknown
Warning  Failed   Error: ImagePullBackOff
----

The image tag `9.9` doesn't exist — the registry can't find it. Let's confirm the image reference:

[source,bash,role="execute"]
----
oc get deployment weather-frontend -n debug-lab -o jsonpath='{.spec.template.spec.containers[0].image}'
----

**Fix**

Update the image to the correct tag (`latest`):

[source,bash,role="execute"]
----
oc set image deployment/weather-frontend weather-frontend=registry.access.redhat.com/ubi9/httpd-24:latest -n debug-lab
----

**Verify**

[source,bash,role="execute"]
----
oc get pods -l app=weather-frontend -n debug-lab
----

You should see `1/1 Running`.

**Takeaway:** ImagePullBackOff = wrong image name, missing tag, or no pull credentials. Always check `oc describe pod` → Events.

=== Problem 2: CrashLoopBackOff (weather-api)

**Diagnose**

The pod keeps starting and crashing. In the **OCP Console**, click the `weather-api` pod, then the **Logs** tab. The log viewer shows the error message with real-time streaming. If the pod has restarted, use the container dropdown on the left (highlighted in red below) to select the previous container's logs.

image::debugging-api-logs.png[weather-api pod Logs tab showing missing config file error]

You can also check via CLI:

[source,bash,role="execute"]
----
oc logs deployment/weather-api -n debug-lab
----

You should see:

----
Error: Missing required config file /etc/weather/config.yaml
----

The app is trying to read a config file that doesn't exist.

**Fix**

Create the missing config file as a ConfigMap and mount it into the container. We'll also fix the container command to run the actual application:

[source,bash,role="execute"]
----
oc create configmap weather-api-config --from-literal=config.yaml="port: 8080" -n debug-lab
----

[source,bash,role="execute"]
----
oc patch deployment weather-api -n debug-lab --type=json -p='[
  {"op":"add","path":"/spec/template/spec/volumes","value":[{"name":"config","configMap":{"name":"weather-api-config"}}]},
  {"op":"add","path":"/spec/template/spec/containers/0/volumeMounts","value":[{"name":"config","mountPath":"/etc/weather"}]},
  {"op":"replace","path":"/spec/template/spec/containers/0/command","value":["/bin/sh","-c","cat /etc/weather/config.yaml && echo Config loaded successfully && sleep 3600"]}
]'
----

**Verify**

[source,bash,role="execute"]
----
oc get pods -l app=weather-api -n debug-lab
----

You should see `1/1 Running`.

**Takeaway:** CrashLoopBackOff = the app starts then exits. Always check `oc logs` and `oc logs --previous`. Common causes: missing config, bad command, failed health checks, out of memory.

=== Problem 3: Resource Constraint Violation (weather-cache)

**Diagnose**

You may have noticed `weather-cache` doesn't appear in the pod list at all. In the **OCP Console**, navigate to **Workloads -> Deployments** (project: `debug-lab`) and click `weather-cache`. The **Events** tab immediately shows the `FailedCreate` error explaining why no pod was created.

image::debugging-cache-deployment-events.png[weather-cache deployment Events showing FailedCreate due to LimitRange violation]

Let's also check via CLI:

[source,bash,role="execute"]
----
oc get deployment weather-cache -n debug-lab
----

It shows `0/1` ready - the pod was never created. When a pod can't even be created, check the **ReplicaSet** events:

[source,bash,role="execute"]
----
oc describe replicaset -l app=weather-cache -n debug-lab | tail -5
----

You should see:

----
Warning  FailedCreate  ...  maximum memory usage per Container is 2Gi, but limit is 1Ti
----

A `LimitRange` in the namespace caps containers at 2Gi memory, but the deployment requests `512Gi` (a typo - should be `512Mi`). Let's confirm:

[source,bash,role="execute"]
----
oc get deployment weather-cache -n debug-lab -o jsonpath='{.spec.template.spec.containers[0].resources}' | python3 -m json.tool
----

[source,bash,role="execute"]
----
oc get limitrange -n debug-lab -o yaml | grep -A4 "max:"
----

**Fix**

Fix the memory typo - change `512Gi` to `512Mi` and `1Ti` to `1Gi`:

[source,bash,role="execute"]
----
oc set resources deployment/weather-cache -n debug-lab --requests=cpu=100m,memory=512Mi --limits=cpu=500m,memory=1Gi
----

**Verify**

[source,bash,role="execute"]
----
oc get pods -l app=weather-cache -n debug-lab
----

You should see `1/1 Running`. The pod can now be created because the resource requests fit within the LimitRange.

**Takeaway:** When a pod never appears at all, check the ReplicaSet events (`oc describe replicaset`). LimitRanges and ResourceQuotas can silently prevent pods from being created.

=== Problem 4: CreateContainerConfigError (weather-backend)

**Diagnose**

[source,bash,role="execute"]
----
oc describe pod -l app=weather-backend -n debug-lab | grep -E "Warning|Error|configmap"
----

You should see:

----
Warning  Failed  Error: configmap "weather-config" not found
----

The deployment references a ConfigMap that doesn't exist:

[source,bash,role="execute"]
----
oc get deployment weather-backend -n debug-lab -o jsonpath='{.spec.template.spec.containers[0].envFrom}' | python3 -m json.tool
----

**Fix**

Create the missing ConfigMap with the keys the app expects:

[source,bash,role="execute"]
----
oc create configmap weather-config \
  --from-literal=API_ENDPOINT="http://weather-api:8080" \
  --from-literal=CACHE_TTL="300" \
  -n debug-lab
----

TIP: You can verify the ConfigMap in the console at **Workloads > ConfigMaps** (project: `debug-lab`) - click `weather-config` to see its keys and values.

Restart the pod to pick up the new ConfigMap:

[source,bash,role="execute"]
----
oc delete pod -l app=weather-backend -n debug-lab
----

**Verify**

[source,bash,role="execute"]
----
oc get pods -l app=weather-backend -n debug-lab
----

You should see `1/1 Running`. Confirm the environment variables are injected:

[source,bash,role="execute"]
----
oc exec deployment/weather-backend -n debug-lab -- env | grep -E "API_ENDPOINT|CACHE_TTL"
----

You should see:

----
API_ENDPOINT=http://weather-api:8080
CACHE_TTL=300
----

TIP: In the console, click the weather-backend pod and select the **Environment** tab to see the injected variables without shelling into the container.

**Takeaway:** CreateContainerConfigError = a referenced ConfigMap or Secret doesn't exist. Check `oc describe pod` for "not found" messages. Create the missing resource and restart the pod.

=== Problem 5: Service Routing Broken (weather-proxy)

**Diagnose**

This pod is `Running` - so what's wrong? In the **OCP Console**, navigate to **Networking -> Services** (project: `debug-lab`) and click `weather-proxy`. The **Pods** tab shows **0 Pods** - immediately revealing that the service can't find any matching pods.

image::debugging-proxy-service-no-pods.png[weather-proxy Service showing 0 Pods due to selector mismatch]

Check via CLI as well:

[source,bash,role="execute"]
----
oc get endpointslice -l kubernetes.io/service-name=weather-proxy -n debug-lab
----

You should see `ENDPOINTS` is empty (`<unset>`). The service has no backends.

Compare the service selector with the pod labels:

[source,bash,role="execute"]
----
echo "=== Service selector ===" && oc get svc weather-proxy -n debug-lab -o jsonpath='{.spec.selector}' | python3 -m json.tool && echo "=== Pod labels ===" && oc get pod -l app=weather-proxy -n debug-lab -o jsonpath='{.items[0].metadata.labels}' | python3 -m json.tool
----

Spot the difference? The service selector says `weather-proxi` (typo) but the pod label is `weather-proxy`.

**Fix**

Patch the service selector to match the actual pod label:

[source,bash,role="execute"]
----
oc patch svc weather-proxy -n debug-lab -p '{"spec":{"selector":{"app":"weather-proxy"}}}'
----

**Verify**

[source,bash,role="execute"]
----
oc get endpointslice -l kubernetes.io/service-name=weather-proxy -n debug-lab
----

You should now see the pod's IP in the `ENDPOINTS` column. Traffic will flow to the pod.

**Takeaway:** When a pod is Running but the service isn't working, check if the service selector matches the pod labels. A single character typo can break routing. Use `oc get endpointslice` to verify.

=== Problem 6: Permission Denied (weather-db)

**Diagnose**

Check the pod status:

[source,bash,role="execute"]
----
oc get pods -l app=weather-db -n debug-lab
----

You'll see `weather-db` in CrashLoopBackOff. Check the logs:

[source,bash,role="execute"]
----
oc logs deployment/weather-db -n debug-lab
----

The log shows: `mkdir: cannot create directory '/var/lib/pgsql': Permission denied`. The container is trying to create a data directory but doesn't have permission.

This is a **Security Context Constraint (SCC)** issue. Check which SCC the pod is running under:

[source,bash,role="execute"]
----
oc get pod -l app=weather-db -n debug-lab -o jsonpath='{.items[0].metadata.annotations.openshift\.io/scc}'
----

It shows `restricted-v2`. This SCC assigns a random user ID to the container for security - the container does NOT run as root. The directory `/var/lib/pgsql` is owned by root inside the image, so the random UID can't write to it.

This is the most common source of pod failures on OpenShift. Containers that expect to run as root will fail under `restricted-v2`.

**Fix**

Mount an `emptyDir` volume at the data path. The container can always write to emptyDir volumes regardless of its UID:

[source,bash,role="execute"]
----
oc patch deployment weather-db -n debug-lab --type=json -p='[
  {"op":"add","path":"/spec/template/spec/volumes","value":[{"name":"db-data","emptyDir":{}}]},
  {"op":"add","path":"/spec/template/spec/containers/0/volumeMounts","value":[{"name":"db-data","mountPath":"/var/lib/pgsql"}]}
]'
----

**Verify**

[source,bash,role="execute"]
----
oc get pods -l app=weather-db -n debug-lab
----

The pod should now be `Running`. Check the logs to confirm the database initialized:

[source,bash,role="execute"]
----
oc logs deployment/weather-db -n debug-lab
----

You should see `Database ready on port 5432`.

**Takeaway:** When you see `Permission denied` in a container log, check the SCC with `oc get pod -o jsonpath='{.metadata.annotations.openshift\.io/scc}'`. Under `restricted-v2`, containers run as a random UID and can't write to root-owned directories. Use `emptyDir` volumes for writable paths, or use images designed to run as non-root.

=== Quick Reference

[cols="1,2,2"]
|===
|Status |Meaning |First Check

|**ImagePullBackOff**
|Can't pull container image
|`oc describe pod` → Events → Check image name/tag

|**CrashLoopBackOff**
|Container starts then crashes
|`oc logs <pod> --previous` → Check app logs

|**CreateContainerConfigError**
|ConfigMap/Secret missing
|`oc describe pod` → Events → "not found" message

|**No pod created**
|LimitRange/ResourceQuota violation
|`oc describe replicaset` → Events → Check resource requests

|**Pending**
|Can't schedule pod
|`oc describe pod` → Events → Check resources/node capacity

|**Running but not working**
|Service selector mismatch or app bug
|`oc get endpointslice` + `oc logs <pod>`

|**Permission denied in logs**
|SCC restricts container UID
|Check SCC: `oc get pod -o jsonpath='{.metadata.annotations.openshift\.io/scc}'` → Mount `emptyDir` for writable paths

|**PVC Pending**
|Storage can't be provisioned
|`oc describe pvc` → Events → Check StorageClass name/existence
|===

== Cleanup & Summary

=== Summary

You diagnosed and fixed six common OpenShift issues:

* **ImagePullBackOff** - wrong image tag (`9.9` doesn't exist) → `oc set image`
* **CrashLoopBackOff** - missing config file → create ConfigMap + volume mount
* **Resource constraint** - memory typo exceeded LimitRange → `oc set resources`
* **CreateContainerConfigError** - missing ConfigMap → `oc create configmap`
* **Service selector mismatch** - typo in selector → `oc patch svc`
* **SCC permission denied** - container can't write to root-owned directory → mount `emptyDir` volume

The pattern is always the same: **check status → describe pod/replicaset → read logs/events → fix → verify.**
