== Runtime Security and Immutable Workloads

In production Kubernetes environments, strict control over what runs inside your cluster is essential for security. Monitoring for file integrity and anomalous behavior helps you detect drift from approved images and configurations.

RHACS includes built-in policies that detect and block risky behaviors such as unauthorized access attempts, persistent threats, lateral movement, and file integrity violations. These policies help you enforce the immutable container principle -- containers should only run exactly what you approved, with no unexpected changes.

== Use Policy-as-Code to Block Anomalous Behavior at Runtime

Package managers such as apt, apk, yum, and dnf are useful on virtual machines, but they have no place in production containers. Allowing them enables uncontrolled software installation, increases the attack surface in your cluster, and breaks auditability and immutability -- core expectations for production workloads.

*Policy-as-Code is an effective way to scale this protection across your clusters.*

Let's look at an example of how we can use Policy-as-Code to block anomalous behavior at runtime.

.Procedure
. *Apply the following policy configuration into the terminal*. It will create a security policy to enforce runtime protection against package manager execution. 
+
[source,bash,subs="+macros,+attributes",role=execute]
----
oc apply -f - <<EOF
---
apiVersion: config.stackrox.io/v1alpha1
kind: SecurityPolicy
metadata:
  name: alpine-linux-package-manager-execution-patient-portal-enforcement
  namespace: stackrox
spec:
  policyName: Alpine Linux Package Manager Execution - Patient-Portal Enforcement
  description: Alert when the Alpine Linux package manager (apk) is executed at runtime
  rationale: Use of package managers at runtime indicates that new software may be being introduced into containers while they are running.
  remediation: Run 'apk --purge del apk-tools' in the image build for production containers. Change applications to no longer use package managers at runtime, if applicable.
  categories:
    - Anomalous Activity
    - Package Management
  lifecycleStages:
    - RUNTIME
  eventSource: DEPLOYMENT_EVENT
  severity: LOW_SEVERITY
  enforcementActions:
    - FAIL_KUBE_REQUEST_ENFORCEMENT
    - KILL_POD_ENFORCEMENT
  policySections:
    - policyGroups:
        - fieldName: Process Name
          booleanOperator: OR
          values:
            - value: apk
            - value: apk-tools
            - value: apk update
            - value: apk add
            - value: apk remove
            - value: apk upgrade
            - value: apk del
            - value: apk cache
            - value: apk clean
            - value: apk info
            - value: apk show
            - value: apk version
  mitreAttackVectors:
    - tactic: TA0011
      techniques:
        - T1105
  criteriaLocked: false
  mitreVectorsLocked: false
  isDefault: false
EOF
----
+
====
Next, you will use `tmux` to watch OpenShift events while running the test so you can see how RHACS enforces the policy at runtime.
====
+

+
[source,sh,role=execute]

----
+
. Next, run a watch on OpenShift events in the first shell pane:
+

----
+
. Press *Ctrl+b, o* to switch to the next pane. (Ctrl+b THEN o)
+
[IMPORTANT]
====
`Ctrl+b` is the prefix key for tmux. `o` is the command to switch to the next pane.
====
+
. Exec into our *patient-frontend* application by getting the pod details and adding them to the following command.
+
[source,sh,role=execute]
----

oc exec -n patient-portal $POD -i --tty -- /bin/sh
----
+

== Violations and Reporting for Audit and Compliance

Event audits support security best practices and give your team a clear record of what happened when a policy fired. RHACS captures forensic detail for each violation so you can investigate, report, and tune policies with confidence.

*Procedure*

. Switch back to the RHACS console in your browser and navigate to the *Violations* page.
. Filter by the policy violation *Policy*, *Name*, `r/alpine` OR by the most recent policy violations (sort by time). You will see a policy violation that has been enforced.
+
image::acs-security-08.png[RHACS Violations list filtered by policy, link=self, window=blank, width=100%]
+
. Click on your *Alpine Linux Package Manager Execution - Patient-Portal Enforcement* to view the details.
+
image::acs-security-09.png[RHACS violation detail for package manager policy, link=self, window=blank, width=100%]
+
[IMPORTANT]
====
If configured properly, each violation record should be pushed to a Security Information and Event Management (SIEM) application and is available to be retrieved via the API. The forensic data shown in the UI is recorded, including the timestamp, process user IDs, process arguments, process ancestors, and enforcement action.
